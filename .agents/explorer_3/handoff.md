## 2. Logic Chain

### 2.1 AndroidManifest.xml — Exact Final Content

The existing manifest already has `xmlns:tools` declared. We add permissions before `<application>`, and service+receivers inside `<application>` after the activity.

**Critical: `<property>` tag for specialUse** — On Android 14+ (API 34), `foregroundServiceType="specialUse"` requires a `<property>` child element explaining the use case. Without it, the service will crash on API 34+ devices at runtime with a `MissingForegroundServiceTypeException`. Since targetSdk is 36, this is mandatory.

**`tools:targetApi="31"`** — The `dataExtractionRules` attribute was introduced in API 31. Although this attribute is already present in the template, adding `tools:targetApi="31"` on `<application>` silences lint warnings for minSdk 24. However, the template generated by Android Studio already handles this gracefully, and since it works without it, we should **not** add it to avoid unnecessary clutter. The existing template is fine as-is.

|---|---|
| `CAMERA` permission | Required for `CameraManager.setTorchMode()` |
| `VIBRATE` permission | Required for haptic feedback via `Vibrator` |
| `FOREGROUND_SERVICE` | Required on API 28+ for `startForeground()` |

### 2.2 BootReceiver.kt — Complete Class Structure

**File**: `app/src/main/java/com/example/mytorch/BootReceiver.kt`

```
Package: com.example.mytorch

Class: BootReceiver : BroadcastReceiver()

    override fun onReceive(context: Context, intent: Intent?)
        1. Guard: if (intent?.action != Intent.ACTION_BOOT_COMPLETED) return
        2. Read SharedPreferences:
             val prefs = context.getSharedPreferences(
                 TorchConstants.PREFS_NAME,    // "mytorch_prefs"
                 Context.MODE_PRIVATE
             )
             val serviceEnabled = prefs.getBoolean(
                 TorchConstants.KEY_SERVICE_ENABLED,  // "service_enabled"
                 false  // default: don't start on first boot
             )
        3. If serviceEnabled:
             val serviceIntent = Intent(context, TorchService::class.java)
             ContextCompat.startForegroundService(context, serviceIntent)

    Notes:
    - Uses ContextCompat.startForegroundService() for API compat (works on API 24+, internally calls startForegroundService on API 26+)
    - Default false: service only auto-starts if user explicitly enabled it
    - onReceive runs on the main thread and must complete quickly — just start the service and exit
```

#### BootReceiver Imports
```kotlin
import android.content.BroadcastReceiver
import android.content.Context
import android.content.Intent
import androidx.core.content.ContextCompat
```

#### API Considerations
- `BOOT_COMPLETED` is delivered after the device finishes booting
- On Android 10+ (API 29), background activity launch restrictions exist, but starting a foreground service from a BroadcastReceiver is still allowed
- On Android 12+ (API 31), there are foreground service launch restrictions from background, **but** `BOOT_COMPLETED` is an exempted broadcast — the system explicitly allows FGS starts from it
- On Android 15 (API 35), short-lived FGS restrictions exist, but a persistent always-on service is fine since it's genuinely long-running

### 2.3 NotificationActionReceiver.kt — Complete Class Structure

**File**: `app/src/main/java/com/example/mytorch/NotificationActionReceiver.kt`

```
Package: com.example.mytorch

Class: NotificationActionReceiver : BroadcastReceiver()

    override fun onReceive(context: Context, intent: Intent?)
        1. Guard: if (intent?.action != TorchConstants.ACTION_TOGGLE_TORCH) return
        2. Create service intent:
             val serviceIntent = Intent(context, TorchService::class.java).apply {
                 action = TorchConstants.ACTION_TOGGLE_TORCH
             }
        3. Start service with toggle command:
             ContextCompat.startForegroundService(context, serviceIntent)

    Notes:
    - The receiver simply relays the toggle command to TorchService
    - TorchService.onStartCommand() must check for ACTION_TOGGLE_TORCH and call TorchManager.toggle()
    - Using startForegroundService because the service may already be running (it will just receive a new onStartCommand)
```

#### NotificationActionReceiver Imports
```kotlin
import android.content.BroadcastReceiver
import android.content.Context
import android.content.Intent
import androidx.core.content.ContextCompat
```

#### How TorchService Creates the Notification Toggle PendingIntent

This is the **call-site** in TorchService (the other team's responsibility) but the contract must be defined here:

```kotlin
// Inside TorchService notification builder:
val toggleIntent = Intent(this, NotificationActionReceiver::class.java).apply {
    action = TorchConstants.ACTION_TOGGLE_TORCH
}
val togglePendingIntent = PendingIntent.getBroadcast(
    this,
    0,                              // requestCode
    toggleIntent,
    PendingIntent.FLAG_UPDATE_CURRENT or PendingIntent.FLAG_IMMUTABLE
)
// Then: .addAction(R.drawable.ic_..., "Toggle", togglePendingIntent)
```

**PendingIntent.FLAG_IMMUTABLE** — Mandatory on API 31+ (Android 12). Since minSdk is 24, always use `FLAG_UPDATE_CURRENT or FLAG_IMMUTABLE`. This is safe on older APIs (the flag is simply ignored pre-31) and required on 31+.

## 3. Cross-Component Integration Map

### 3.1 SharedPreferences Flow

```
┌──────────────┐     write      ┌──────────────────┐     read       ┌──────────────┐
│  MainActivity │ ──────────────▶│  SharedPreferences │◀──────────────│  BootReceiver │
│              │                │  "mytorch_prefs"   │               │              │
│  (UI toggle  │                │                    │               │  (on BOOT_   │
│   switch)    │                │  key:              │               │   COMPLETED) │
│              │                │  "service_enabled" │               │              │
└──────────────┘                └──────────────────────┘              └──────────────┘
```

- **Writer**: MainActivity — when user toggles the "Background Service" switch
- **Reader**: BootReceiver — at boot time to decide whether to start TorchService
- **File**: `"mytorch_prefs"` (Context.MODE_PRIVATE)
- **Key**: `"service_enabled"` (Boolean, default `false`)

### 3.2 Intent Action Flow

```
┌────────────────────────┐   ACTION_TOGGLE_TORCH   ┌──────────────┐
│ NotificationActionRcvr │ ────────────────────────▶│ TorchService │
│ (receives broadcast    │   (relayed via           │ (onStart     │
│  from notification     │    startForeground       │  Command     │
│  PendingIntent)        │    Service intent)       │  handles it) │
└────────────────────────┘                          └──────────────┘
```

- NotificationActionReceiver receives the broadcast and forwards to TorchService via `startForegroundService` with `action = ACTION_TOGGLE_TORCH`
- TorchService checks `intent?.action` in `onStartCommand()` and toggles if it matches

### 3.3 Torch State Broadcast Flow

```
┌──────────────┐   ACTION_TORCH_STATE_CHANGED    ┌──────────────┐
│ TorchService │ ───────────────────────────────▶│  MainActivity │
│ (torch       │   (LocalBroadcast or           │  (updates UI  │
│  toggled)    │    companion StateFlow)         │   state)      │
└──────────────┘                                 └──────────────┘
```

Two viable approaches for torch state → UI:
1. **`LocalBroadcastManager`** — deprecated but functional; register/unregister in onStart/onStop
2. **Companion `StateFlow`** (recommended) — `TorchService.Companion.torchState: MutableStateFlow<Boolean>` — MainActivity collects it in Compose. Simpler, type-safe, no registration lifecycle issues.

**Recommendation**: Use companion `StateFlow` approach. It avoids deprecated APIs, is simpler, and plays naturally with Compose's `collectAsState()`.

## 4. API Level Compatibility Details

### 4.1 Permission Requirements by API Level

| Permission | Required From | Notes |
|---|---|---|
| `CAMERA` | API 1+ (always) | Normal permission for flash; no runtime request needed for torch-only use |
| `VIBRATE` | API 1+ (always) | Normal permission |
| `FOREGROUND_SERVICE` | API 28+ (P) | Normal permission, auto-granted |
| `FOREGROUND_SERVICE_SPECIAL_USE` | API 34+ (U) | Normal permission, auto-granted; **must** be declared in manifest |
| `RECEIVE_BOOT_COMPLETED` | API 1+ (always) | Normal permission |
| `WAKE_LOCK` | API 1+ (always) | Normal permission |
| `POST_NOTIFICATIONS` | API 33+ (Tiramisu) | **Runtime permission** — must request at runtime via `ActivityCompat.requestPermissions()` |

### 4.2 Foreground Service Type Timeline

| API Level | Requirement |
|---|---|
| 24-25 (N) | `startForegroundService()` not available; use `startService()` + `startForeground()`. **But** `ContextCompat.startForegroundService()` handles this transparently. |
| 26-27 (O) | `startForegroundService()` introduced; must call `startForeground()` within 5 seconds |
| 28 (P) | `FOREGROUND_SERVICE` permission required |
| 29-33 | No change for our use case |
| 34+ (U) | `foregroundServiceType` must be declared; `specialUse` requires `FOREGROUND_SERVICE_SPECIAL_USE` permission and `<property>` tag |
| 35+ (V) | Short-lived FGS restrictions — not an issue for always-on service |

### 4.3 PendingIntent Flags

| API Level | Flag Requirement |
|---|---|
| 24-30 | `FLAG_UPDATE_CURRENT` is sufficient |
| 31+ (S) | Must specify `FLAG_IMMUTABLE` or `FLAG_MUTABLE`; crash if neither specified |

**Safe universal pattern**: `PendingIntent.FLAG_UPDATE_CURRENT or PendingIntent.FLAG_IMMUTABLE`
- `FLAG_IMMUTABLE` value (0x04000000) is ignored on API < 31
- On API 31+ it is required — without it, `IllegalArgumentException` is thrown

### 4.4 Service Start Method

Use `ContextCompat.startForegroundService(context, intent)` everywhere (both BootReceiver and NotificationActionReceiver):
- On API 26+: calls `context.startForegroundService(intent)`
- On API 24-25: calls `context.startService(intent)`
- This is provided by `androidx.core:core-ktx` which is already a dependency

### 4.5 BootReceiver — Background FGS Launch Exemptions

On Android 12 (API 31+), starting a foreground service from the background is restricted. However, `BOOT_COMPLETED` is an **exempted broadcast** that explicitly allows FGS starts. Our BootReceiver is safe.

On Android 14+ (API 34), the BOOT_COMPLETED exemption still applies. No additional handling needed.

## 5. Caveats

1. **CAMERA permission for torch**: On API 23+ (runtime permissions model), `CAMERA` is a dangerous permission. However, `CameraManager.setTorchMode()` does NOT require the CAMERA runtime permission — it only needs the camera service to be available. The `<uses-permission android:name="android.permission.CAMERA" />` in manifest is technically optional for torch-only use. The implementation plan includes it, so we keep it, but the Worker should be aware that no runtime request is needed for torch-only functionality. If the app only controls the flash LED, consider removing this permission. **Recommendation**: Keep it per the plan to be safe, but do NOT request it at runtime.

2. **`tools:targetApi` on application element**: The template's `dataExtractionRules` attribute (API 31+) may trigger a lint warning since minSdk is 24. Android Studio templates typically handle this with lint suppress, and it works fine at runtime on older devices (the attribute is simply ignored). If lint complains, add `tools:targetApi="31"` to the `<application>` tag. Not adding it now since the template works without it.

3. **Direct_boot_aware**: We do NOT set `android:directBootAware="true"` on BootReceiver. This means the receiver won't fire during Direct Boot (before the user unlocks the device after a restart). This is acceptable for v1 — the service will start once the user unlocks their phone.

4. **Battery optimization**: On some OEMs (Xiaomi, Samsung, Huawei), the service may be killed despite being a foreground service. This is an OEM-specific issue beyond our control in v1. No manifest changes address this.

5. **ACTION_TORCH_STATE_CHANGED**: If the team uses companion `StateFlow` instead of broadcasts for torch state → UI communication, this constant becomes unused. Include it anyway for potential future use or remove if the StateFlow approach is confirmed.

## 6. Conclusion

The manifest modifications are straightforward with one **critical detail**: the `<property>` tag inside `<service>` for the `specialUse` foreground service type is mandatory on API 34+ and will cause a runtime crash if omitted.

Both receivers are simple relay classes:
- **BootReceiver**: reads a boolean preference and conditionally starts the service
- **NotificationActionReceiver**: receives a broadcast from the notification PendingIntent and relays a toggle command to the service

The **TorchConstants** object is the integration linchpin — all shared strings (preference keys, action strings, channel IDs, notification ID) must live here and be referenced by every component. This eliminates the most common integration failure mode: string mismatches.
